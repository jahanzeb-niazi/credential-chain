UC-01: Issue Academic Credentials
Use Case ID: UC-01
Use Case Name: Issue Academic Credentials
Actors: University Administrator
Type: Primary
Description:
The university administrator logs into CredLedger and fills in a form with the student's credential
details such as name, degree type, and graduation date. The system uploads this information to
IPFS and gets back a Content Identifier (CID). The CID is then recorded on the Ethereum
blockchain through a smart contract call. Once the transaction is confirmed, the credential is
officially issued and can be verified by anyone on the network.

UC-02: Manage Existing Credentials
Use Case ID: UC-02
Use Case Name: Manage Existing Credentials
Actors: University Administrator
Type: Primary
Description:
The university administrator logs into the system and opens the credential management
dashboard. The admin can search for credentials by student name or ID and view the full list of
issued credentials for enrolled or graduated students. The system displays the current status of
each credential including whether it is active or revoked. The admin can select any record to
view its full details and take further actions such as updating or revoking it.

UC-03: Revoke Credentials
Use Case ID: UC-03
Use Case Name: Revoke Credentials
Actors: University Administrator
Type: Primary
Description:
The university administrator identifies a credential that was issued fraudulently or contains an
error and navigates to it in the dashboard. The admin selects the revoke option, enters a reason,
and confirms the action. The system sends a revocation transaction to the Ethereum smart
contract, which permanently marks the credential as revoked on-chain. Any future verification
attempt on this credential will return a revoked status and the revocation timestamp is stored for
audit purposes.

BUC-04: Update Credential Data
Use Case ID: UC-04
Use Case Name: Update Credential Data
Actors: University Administrator
Type: Primary
Description:
When an administrative correction is needed on an existing credential such as a spelling error in
the student name or an incorrect graduation date, the administrator locates the record and submits
the updated information. The system uploads the corrected metadata to IPFS, generates a new
CID, and updates the reference on-chain through a smart contract call. The old record is
preserved on the blockchain for audit purposes while the new CID reflects the corrected data.

UC-05: On-chain CID Generation
Use Case ID: UC-05
Use Case Name: On-chain CID Generation
Actors: University Administrator
Type: Primary
Description:
When a credential is created or updated, the system automatically stores the credential metadata
on IPFS through the application layer. IPFS returns a unique Content Identifier (CID) based on
the file content. The system then sends a transaction to the Ethereum blockchain to record this

CID alongside the credential information. This process ensures that the metadata is tamper-
resistant since any change to the content would produce a completely different CID.

UC-06: View Owned Credentials
Use Case ID: UC-06
Use Case Name: View Owned Credentials
Actors: Student / Graduate
Type: Primary
Description:
A student logs into CredLedger by connecting their MetaMask wallet. The system fetches all
credentials linked to their wallet address from the blockchain and displays them on a personal
dashboard. The student can see each credential's details including the degree name, institution,
graduation date, and current status. This gives the student a permanent and portable record of all
their earned academic qualifications.

UC-07: Generate Verification Links
Use Case ID: UC-07
Use Case Name: Generate Verification Links
Actors: Student / Graduate
Type: Primary
Description:
A student selects a credential from their dashboard and clicks the option to generate a
verification link. The system creates a unique secure link tied to that credential on the blockchain
and also provides a QR code version. The student copies the link or downloads the QR code and
shares it with a prospective employer or institution. The recipient can use the link to check
authenticity without contacting the university and without seeing any data beyond what the
student chose to share.

UC-08: Share Cryptographic Proofs
Use Case ID: UC-08
Use Case Name: Share Cryptographic Proofs
Actors: Student / Graduate
Type: Primary
Description:
Instead of sharing the full credential document, a student can choose to share a cryptographic
proof that they hold a valid degree without revealing sensitive personal details. The system
generates a proof derived from the on-chain credential data that a verifier can check against the
blockchain. This allows the student to prove authenticity to a third party while maintaining
control over how much personal information is disclosed.

UC-09: Direct Blockchain Validation
Use Case ID: UC-09
Use Case Name: Direct Blockchain Validation
Actors: Verifier
Type: Primary
Description:
A verifier checks a credential by querying the Ethereum blockchain directly through the
CredLedger interface, without needing to contact or trust the issuing university. The system calls
the smart contract with the credential identifier and retrieves the on-chain record. The verifier
sees whether the credential is valid and gets all associated details straight from the immutable
ledger. This removes any dependency on a third party and makes the verification process fully
trustless.

UC-10: Scan QR Codes
Use Case ID: UC-10
Use Case Name: Scan QR Codes
Actors: Verifier
Type: Primary
Description:
A verifier receives a physical or digital diploma that contains a QR code. The verifier opens the
CredLedger web application and uses the QR scanner feature to scan the code. The system
decodes the QR code to extract the credential identifier and queries the blockchain to retrieve the
credential details. The result is displayed on screen within seconds, confirming whether the
credential is valid, revoked, or not found in the system.

UC-11: Audit History Review
Use Case ID: UC-11
Use Case Name: Audit History Review
Actors: Verifier
Type: Primary
Description:
A verifier checks the full lifecycle of a credential by accessing its history on the blockchain. The
system retrieves and displays all recorded events for that credential such as when it was issued,
whether it was ever updated, and whether it has been revoked. This gives the verifier a complete
and tamper-proof picture of the credential from the moment it was created to its current state,
helping them make more informed decisions.

UC-12: Timestamp Verification
Use Case ID: UC-12
Use Case Name: Timestamp Verification
Actors: Verifier
Type: Primary
Description:
A verifier wants to confirm exactly when a degree was issued and recorded on the blockchain.
The system retrieves the Ethereum block timestamp associated with the credential issuance
transaction and displays it clearly. Since blockchain timestamps are immutable and cannot be
altered retroactively, the verifier can be confident that the date shown is the exact time the
credential was officially recorded, which rules out the possibility of backdating.

UC-13: Onboard Regulators
Use Case ID: UC-13
Use Case Name: Onboard Regulators
Actors: Government
Type: Primary
Description:
The government authority logs into the CredLedger system as the root actor and adds a new
regulatory body to the ecosystem. This involves registering the regulator's Ethereum wallet
address and details through a smart contract call. Once added, the regulator gains the authority to
manage the registry of accredited institutions and enforce issuance policies. The government
maintains ultimate control and can remove or suspend regulators if necessary.

UC-14: Authorize Institutions
Use Case ID: UC-14
Use Case Name: Authorize Institutions
Actors: Regulator
Type: Primary
Description:
A regulator logs into the CredLedger admin panel and submits a request to add a new university
to the list of accredited institutions. The system calls the relevant smart contract to register the
institution's Ethereum wallet address and official details on-chain. Once confirmed, the
institution can start issuing and managing credentials through the system. The regulator can view
and manage the full registry of authorized institutions at any time.
UC-15: Audit Institutional Activity
Use Case ID: UC-15
Use Case Name: Audit Institutional Activity
Actors: Regulator
Type: Primary
Description:
A regulator selects an institution from the registry and reviews its full on-chain activity history.
The system retrieves all blockchain events related to that institution including credentials issued,
updated, and revoked, along with timestamps for each action. The regulator uses this information
to check whether the institution is following the required policies. If any violations are found, the
regulator can flag the institution or suspend their credential issuance rights.
